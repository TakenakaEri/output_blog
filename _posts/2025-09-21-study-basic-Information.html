システム開発

開発手法

  • RAD（Rapid Application Development）
    プロトタイプモデルとスパイラルモデルを掛け合わせてような感じ
    エンドユーザーと開発者による少数構成のチームを組み、開発支援ツールなどを使用して短期間で開発を行うことを重視。
    タイムボックス：開発の期限を設けてRADを実施すること
  • アジャイル（XP）
    スパイラルモデルの派生系（より短い反復単位。週単位）

    XP（eXtreme Programming）がアジャイルの主な開発手法

    • テスト駆動開発
    • ペアプログラミング
    • リファクタリング
    • ソースコードの共同所有
      コードの所有者に断りなく変更でき、責任を負う
    • 継続的インテフラーション
      単体テストお終えたプログラムはすぐに結合して、結合テストを行う
    • YAGNI（You Are't Going to Need it.）
      今必要とされている機能だけのシンプルな実装に留める

リバースエンジニアリング

既存のソフトウェアの動作を分析することでソースコードを導き出す。
既にあるソフトウェアを再利用することが目的。

フォワードエンジニアリング

リバースエンジニアリングによって得られた仕様をもとに新しいソフトウェアを開発する手法

マッシュアップ

公開されている複数のサービスを組み合わせることによって新しいサービスを作り出すこと

スプリントで実施するイベント

  • スプリントプランニング
    スプリントの開始前に策定されるスプリントの計画
  • スプリントレトロスペクティブ
    スプリントの終了後に実施される振り返り（評価と改善）
  • スプリントレビュー
    スプリントの終了後にチームメンバーとすテークホルダが成果物を評価し意見を言い合う
  • デイリースクラム

DFD（Date Flow Diagram）

データの流れを図として表したもの

  • プロセス（処理）
    ○を使って、データ変換の処理をあらわす
    例）受注や在庫確認など
  • データの源泉と吸収
    ⬜︎を使って、データの発生元や最終的な行き先をあらわす
  • データフロー
    →を使って、データの流れを表す。フローの上にはデータ名を書く
  • データストア
    二重線をつかってファイルやデータベースなど、データを保存する場所を表す

ER-図

  • 実態（エンティティ）：資格で囲った部分
  • 関連（リレーションシップ）：矢印で繋ぐ
  • UML
    UMLはシステム延滞の設計に使用する。ER図はDB設計に使用する

チェックディジット

誤入力を判定するためにコードへ付加された数字。

コードの種類

  • 順番コード
    連続した番号を順番に付与していく
  • 区分コード
    いくつかのグループに分け、そのグループごとに連続した番号を付与する
  • けた別コード
    けたごとに意味を持たせたコード。
    大分類・中分類・小分類といった意味を桁に付与して階層かすることができる
    例）頭2桁は支店コード、次の2桁た部署コードなど
  • 表意コード（ニモニックコード）
    項目の意味を表す略称や記号などにより表現する
    例）日本はJPとか、黒の32GのUSBはUSB-B-32Gとか

入力ミスを判定するチェック方法

  • ニューメリックチェック
    数値として扱う必要のあるデータに、文字などを数値として扱えないものが含まれていないかをチェックする
  • シーケンスチェック
    対象とするデータが一定の順序で並んでいるかをチェックする
  • リミットチェック
    データが適正な範囲内にあるかをチェックする
  • フォーマットチェック
    データの形式がフォーマット通りかをチェックする
  • 照合チェック
    入力されたコードが表中に登録されているかを照合する（登録済みでないコードの入力を避けるため）
  • 論理チェック
    販売数と在庫数と仕入数の関係など、対となる項目の値に矛盾がないかチェックする
  • 重複チェック

プログラムの構造化設計

各プログラムをモジュール単位に分解・階層化させることをプログラムの構造化設計

    データの流れに着目
  • STS分割法
    入力処理（Source）・変換処理（Transform）・出力処理（Sink）の3つの構造に分割する
  • トランザクション（TR）分割法
    一連の処理（トランザクション）単位に分割する方法
  • 共通機能分割法
    共通機能をモジュールとして分割する方法

データの構造に着目
深掘りしなくてOK。「人の名前がついている分割法はデータ構造に着目したもの」くらいで大丈夫

  • ジャクソン法
  • ワーニエ法

モジュール強度

どれだけ機能的に特化できているか？高いほど独立性が高くて好ましい

  • 機能的強度
    単一の機能を実行するためのモジュール
  • 情報的強度
    オブジェクト指向のカプセル化。データ構造というなの情報をひとまとめにしている
  • 連絡的強度
    一連の処理をまとめたもの。各機能は共通の入力値もしくは出力値を参照している
  • 手順的強度
    一連の処理をまとめたもの。各機能にデータ的なつながりはない
  • 時間的強度
    初期化や終了処理など、特定の時点（時間）で必要になるもの
  • 論理的強度
    モジュール呼び出し時の引数によってどの機能を実行するかが決まる
  • 暗号的強度
    ただ分けただけのもの

モジュール結合度

モジュールの独立性が高いほど好ましい

  • データ結合
    単一のデータを引数として渡すモジュール
  • スタンプ結合
    データ構造を引数として渡すモジュール
  • 制御結合
    他のモジュールの動作を制御する目的で引数として制御パラメーターを受け渡すモジュール
  • 外部結合
    単一のデータを大域的データ（グローバル領域のデータ）として受け渡すモジュール
  • 共通結合
    データ構造を大域的データ（グローバル領域のデータ）として受け渡すモジュール
  • 内容結合
    他のモジュール内のデータを直接参照するモジュール

補足メモ：制御結合の例 

    def process_data(data, operation_flag):
        if operation_flag == 1:
            return sort_data(data)
        elif operation_flag == 2:
            return filter_data(data)
        elif operation_flag == 3:
            return transform_data(data)
        else:
            return data

    # 呼び出し側
    result = process_data(my_data, 2)  # フィルタ処理を指示

この例では、operation_flagが制御パラメーターとして機能し、process_dataモジュール内部の処理の流れを外部から制御しています。

テストの流れ

  • 単体テスト
  • 結合テスト
  • システムテスト
    • 性能テスト
      処理能力は十分か？
    • 負荷テスト
      高い負荷の下でも問題ないか？
    • 機能テスト
      要求した機能がちゃんと動くか？

ブラックボックステストのテストデータの基準

  • 同値分割
    代表的な値を抜き出してテストデータに用いる
  • 限界値（境界値）分析
    値の境目部分を重点的にチェックする

ホワイトボックステストの網羅基準

  • 命令網羅
    すべての命令を最低1回は通すようにテストする
    命令が一度実行できればYesの分岐を通るテストパターンがあればOK
  • 判定条件網羅（分岐網羅）
    すべての分岐を最低一度は通すようにテストする
    YesとNoの分岐、2つ分のテストパターンが分岐している場合、2つ分のテストパターンが必要
  • 条件網羅
    ここの条件が、真と偽の値を最低1回は満たすようにする
    それぞれで一度ずつ信徒に背をためせていれば良い
  • 複数条件網羅
    複数の条件が取りうる真偽値の全ての組み合わせを網羅するテスト
    とにかく全ての組み合わせを検証する

結合テスト

  • トップダウンテスト
    上位モジュールからテストする
    スタブと呼ばれる仮のモジュールをくっつけてテストする
  • ボトムアップテスト
    ドライバと呼ばれる仮のモジュールをくっつけてテストする
  • その他
    • 折衷テスト
    • ビッグバンテスト

リグレッションテスト（退行テスト）

プログラムを修正した時に、その修正内容がこれまで正常に動作していた範囲に悪影響を与えていないか（新たなバグを誘発しないか）を確認するためのテスト
